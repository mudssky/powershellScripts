# 整理分类方法论

## 选择原则

先选主分类维度，再补辅助标签。主维度回答“这个东西应该去哪里”，辅助标签回答“它有什么状态、风险或检索入口”。

不要让对象同时被多个目录哲学拉扯。若多个方法都合理，优先选择用户最可能查找、维护和执行动作的入口。

## 方法路由表

| 方法 | 适用场景 | 输入信号 | 不适用情况 |
|---|---|---|---|
| 使用场景优先 | 文档、工具、资料库、下载目录 | 用户会按任务查找 | 对象天然有强生命周期或强框架约定 |
| 生命周期分类 | inbox、active、reference、archive、deprecated、generated | 对象状态比主题更重要 | 长期知识库按状态查找会割裂主题 |
| 领域/职责分类 | 业务域、技术域、模块职责 | 对象围绕稳定领域演化 | 领域尚未稳定、只有临时文件 |
| 对象类型分类 | 图片、音频、PDF、脚本、配置 | 类型决定处理工具 | 长期维护时只按扩展名会丢语义 |
| 频率/重要性分类 | 高频入口、低频参考、长期归档 | 打开频率差异大 | 不能替代正式归属规则 |
| LATCH | 信息架构、大纲、资料集合 | 需要选择组织轴 | 不负责判断风险和引用 |
| MECE | 检查分类覆盖与重复 | 分类树混乱或“其他”太多 | 现实对象允许交叉标签时不要硬互斥 |
| 渐进整理 | 文件目录、代码项目、大批迁移 | 移动或删除风险高 | 纯文章大纲调整可能过重 |

## 使用场景优先

按用户完成任务时最自然的入口分类。适合 cheatsheet、操作手册、工具集合、项目文档和资料库。

规则：

- 优先问“使用者会怎么找它”，再问“它是什么类型”。
- 一个对象跨多个主题时，放在最主要的使用入口，并用交叉链接补次要入口。
- 不要因为文档里偶然出现某个命令、库名或关键词就改归属。

## 生命周期分类

按对象状态分类，例如 `inbox`、`active`、`reference`、`archive`、`deprecated`、`generated`。

规则：

- `inbox` 必须有定期清空机制，否则会变成新的垃圾桶。
- `archive` 表示低频保留，不等于可删除。
- `deprecated` 要说明替代入口。
- `generated`、cache、report、build output 应优先进入忽略或产物目录，不应混入源码真相。

## 领域/职责分类

按业务域、技术域、模块职责或团队边界分类。适合长期演进的代码、文档和配置。

规则：

- 领域名优先稳定、可被团队复用。
- 跨领域共享能力应放在共享层，避免复制到多个领域。
- 如果一个领域只有一两个临时对象，先暂缓新建大目录。

## 对象类型分类

按格式或媒介分类，例如 image、audio、video、pdf、script、config。

规则：

- 适合下载目录、素材库和批处理入口。
- 对代码项目和长期文档，类型只能做辅助标签，不能单独决定归属。
- 无扩展名但有明确格式的文件，优先补清晰扩展名。

## 频率/重要性分类

按高频入口、低频参考、长期归档、临时缓存等分类。

规则：

- 高频入口应该短路径、易发现、命名直观。
- 低频参考可以放深一点，但要有索引或链接。
- 重要但低频的内容不要放进 cache/tmp。

## LATCH

LATCH 用 Location、Alphabet、Time、Category、Hierarchy 五个轴选择信息结构。

建议：

- Location：地理、路径、部署环境强相关。
- Alphabet：词典、索引、联系人等查找型内容。
- Time：日志、日记、版本、时间线。
- Category：主题、职责、类型。
- Hierarchy：组织树、章节、权限、依赖层级。

## MECE

MECE 用来检查分类是否尽量互斥且整体覆盖。

规则：

- 同一层级的分类口径要一致。
- “其他”过大时，说明主维度可能选错。
- 现实对象允许交叉标签，不要为了互斥破坏查找路径。

## 渐进整理

适合高风险文件目录和代码项目迁移。

步骤：

1. 盘点对象、入口、引用、生成物、本机私有文件。
2. 先标记归属和风险，不立即移动。
3. 小批量迁移，优先低风险对象。
4. 同步引用、脚本、配置和文档。
5. 执行验证命令或搜索断链。
6. 保留回滚路径。
