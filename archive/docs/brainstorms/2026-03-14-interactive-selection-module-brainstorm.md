# 通用交互选择模块

## What We're Building

为 `psutils` 新增一个专用的交互选择模块，用来承载“优先使用 `fzf`，缺失时自动降级到 PowerShell 文本选择”的统一能力。这个模块不只服务 benchmark 调度脚本，而是作为仓库里所有“从候选项中选择内容”的通用基础设施，供 benchmark、清理脚本以及后续其他命令复用。

本次关注的是“选择体验”的统一，而不是某个单一脚本的局部优化。目标是让调用方只关心“给我这些候选项，帮我选”，不再在各个脚本里分别处理 `fzf` 是否存在、文本降级、多选输入解析等重复逻辑。

## Why This Approach

讨论过三种方向：继续把能力塞进现有 `functions.psm1`、单独新增 `selection.psm1` 模块、以及只在 benchmark 调度脚本附近做局部封装。最终选择专用模块方案，因为这次需求已经明确不是 benchmark 独有能力，而是会被多个脚本复用的交互基础设施。

相比之下，继续堆进 `functions.psm1` 虽然改动更小，但会进一步混杂历史搜索、快捷键绑定和通用交互选择这几类职责；只做局部封装则会在第二个使用者出现时立即产生重复代码和再抽象成本。专用模块的改动面更大，但长期边界最清楚。

## Key Decisions

- 新能力放在 `psutils/modules/` 的专用模块中，而不是继续塞进 `functions.psm1`。
- 公开 API 应支持字符串列表和对象列表两种输入。
- 默认行为是单选；通过显式参数切换到多选。
- 返回值直接返回原始项本身，而不是索引或包装对象。
- 对象输入时，调用方必须显式提供显示逻辑，例如 `DisplayScriptBlock` 或 `DisplayProperty`；不做隐式字段猜测。
- 优先使用 `fzf` 进行交互选择；若当前环境没有 `fzf`，自动降级到文本编号列表 + `Read-Host`。
- 用户取消选择时，函数默认返回 `$null` 或空数组，由调用方自行决定后续处理。

## Resolved Questions

- benchmark 调度脚本不应在缺少参数时只报错列列表，而应该走统一选择能力。
- `fzf` 交互不是唯一目标；更重要的是把“自动降级”的选择体验统一沉到公共模块中。
- 文本降级模式需要支持多选，而不只是最简单的单选。
- 自动推断对象显示字段不是这次需要的复杂度，显式指定更稳妥。
