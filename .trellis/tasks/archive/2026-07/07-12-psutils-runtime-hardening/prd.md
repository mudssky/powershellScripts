# 加固 psutils 运行时安全与健壮性

## Goal

对静态分析和源码审计发现的高风险运行路径逐项验证，只实施有真实故障、安全或可诊断性收益的加固。

## Background

- 高信号候选包括 `Set-SSHKeyAuth` 的 passphrase 处理、历史命令动态执行、多个空 catch、自动变量命名和跨平台路径/命令边界。
- PSScriptAnalyzer 的大量 `Write-Host` 告警主要来自交互式命令和 demo，不能作为批量重写依据。

## Requirements

- 为每个候选问题建立可复现行为或明确风险模型，再决定修复、记录例外或保持现状。
- 敏感值不得无必要地进入命令行、日志、错误或返回对象。
- 动态执行必须是用户显式动作，限定输入来源并清楚记录风险边界。
- 预期可忽略的异常至少提供 `Verbose` 诊断；不可恢复异常不得静默吞掉。
- 自动变量、跨平台命令和路径处理不得依赖当前平台偶然行为。
- 不进行无差别 PSScriptAnalyzer 清零。

## Acceptance Criteria

- [ ] 每个高风险候选都有复现、风险结论和处理决定。
- [ ] 入选修复有聚焦 Pester 测试，覆盖成功、失败和跨平台分支。
- [ ] 敏感参数和动态执行路径有明确的用户可见契约。
- [ ] 被保留的静态分析例外有设计理由，不通过全局禁用隐藏。
- [ ] `pnpm qa` 与 `pnpm test:pwsh:all` 通过。

## Out of Scope

- 清理所有 style/verb/Write-Host 告警。
- 改变明确用于交互输出的命令为纯对象 API，除非测试证明当前行为有问题。
- 与 `psutils` 无关的仓库安全审计。

## Dependencies

- 依赖 `07-12-psutils-core-contract` 稳定入口和导出测试。
- 建议在 API 边界任务之后实施，减少同文件并行修改。
