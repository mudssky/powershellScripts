# 浏览器书签整理工作流

## 推荐步骤

1. 创建 snapshot：解析原始书签，生成只读原始状态。
2. 收集 evidence：统计目录、domain、重复项、空目录、疑似乱码标题和可选链接检测结果。
3. 废弃链接处理：基于 evidence 生成标记、归档或更新 URL 的候选操作，不直接删除。
4. 新目录结构：只生成目录操作，尊重原目录，避免固定分类模板。
5. 链接分配：每批读取目标目录树和一定数量书签，生成批量移动/重命名/归档操作。
6. WebUI 审核：用户可接受、拒绝、修改目标目录，或要求新方案。
7. 导出：CLI replay 已批准 operations，生成新的 bookmarks HTML。

WebUI 是默认入口，不是最后才打开的报告页。`analyze` 完成 snapshot 和 evidence 写入后应后台启动本地临时服务，并在终端输出 `Review UI: http://127.0.0.1:<port>`、PID、日志路径和自动关闭时间。服务默认 1 小时自动关闭；需要只生成文件时才显式使用 `--no-review`，需要调试服务时才使用 `--foreground`。

WebUI 应全程作为状态面板：展示当前目录树、阶段进度、待审核操作、已应用操作和导出预览。CLI 每次应用操作后，WebUI 刷新即可看到新的当前状态。

运行契约：

- stdout 只输出用户/agent 需要立即看到的入口信息，例如 workspace、URL、PID、日志路径、导出文件路径和错误摘要。
- 后台服务内部事件写入 `review-server.log`，使用 Python `logging` 和模块级 `logger = logging.getLogger(__name__)`。
- 时间戳统一通过包内时间封装生成，默认使用系统本地时区，避免业务代码散落 UTC 或固定时区。

## 分类方法论

这些只是方法论，不是固定分类模板。默认应尊重用户原目录，优先复用已有目录；只有在目录过深、混杂、重复、空目录或命名不清时才建议调整。

优先选择一个主维度：

- 用途：工作、学习、资料、工具、娱乐、临时阅读。
- 主题：编程、AI、系统运维、设计、文档、产品、生活。
- 行动状态：待读、常用、参考、归档、失效待删。
- 来源可信度：官方文档、一手资料、社区讨论、博客、工具站。

辅助标签可以是语言、框架、账号体系、年份、项目名或平台名。不要把辅助标签同时展开成多层目录，否则书签会继续变深。

设计目录树时遵守这些原则：

- 原目录优先：能通过重命名、合并、移动少量目录解决的问题，不新建一套完全不同的目录。
- 一次只选一个主轴：不要同时按“主题 / 用途 / 状态 / 来源”并列展开多套目录树。
- 目录深度优先控制在 2 到 3 层；超过 3 层时，先考虑把细分信息放到标题、标签式命名或批次备注里。
- 常用入口靠前，归档和低频资料靠后；“临时”“待读”“待确认”应有清理出口，避免变成永久垃圾桶。
- 官方文档、API Reference、Release Notes 等高可信资料可以单独保留，也可以作为主题目录下的 `官方` 子目录。
- 框架或语言生态书签优先按实际使用场景归类；如果某个生态数量足够多，再独立成目录。
- 私网、公司系统、家庭服务和 Tailscale 地址应保留上下文说明，不要因为当前网络不可达而直接归入死链。

常见目录主轴选择：

| 场景 | 推荐主轴 | 适合的二级目录 |
| --- | --- | --- |
| 编程学习和开发资料较多 | 主题 | 语言、框架、工具、官方文档、问题排查 |
| 工作系统和内部链接较多 | 用途 | 日常入口、项目、团队、运维、账号与管理 |
| 稍后阅读和文章堆积较多 | 行动状态 | 待读、精读、参考、归档、失效待确认 |
| 研究资料和信息源较多 | 来源可信度 | 官方、一手资料、论文/标准、社区、博客 |
| 个人生活服务较多 | 用途 | 财务、健康、旅行、购物、家庭服务 |

可以混合使用，但混合应发生在不同分支内。例如根目录用“用途”，`开发` 分支内再按“主题”细分；不要在根目录同时放一批“用途目录”和一批“主题目录”抢主轴。

## 目录重构步骤

第一步先生成新的目录树草案，只处理目录，不移动具体书签。目录草案应来自原目录上下文、统计报告和用户目标，而不是固定模板。

目录阶段建议输出一组 `tree_restructure` operations：

- `create_folder`：创建确实缺失的新目录。
- `rename_folder`：让含糊、过时或乱码目录变清晰。
- `move_folder`：调整目录层级。
- `merge_folder`：合并语义重复或只有少量内容的目录。
- `delete_empty_folder`：删除空目录候选，仍需用户审核。

目录阶段完成后，使用 CLI replay 当前状态得到新目录树，再进入书签分配阶段。

书签分配阶段一次处理一批链接，例如 30 到 80 个。每批只需要读取当前目录树、这一批书签、分析 evidence 和已有 operations，然后生成 `bookmark_assignment` operations 交给用户审核。这样上下文占用稳定，也方便用户逐批修正方向。

## 常见处理策略

- 重复 URL：保留标题更清晰、目录更合适或最近添加的一项，其他项列为合并候选。
- 死链：先归档或标记，除非用户明确要求删除。
- 重定向：如果最终 URL 稳定且站点可信，可以列为更新 URL 候选。
- 空目录：若无明确未来用途，列为删除候选。
- 疑似乱码标题：先根据 URL/domain 推断标题，必要时联网打开确认。
- 过深目录：优先压平到 2 到 3 层，用标签或标题前缀补充细分信息。
- 局域网/Tailscale/公司内网链接：标记为上下文依赖链接，默认不作为死链候选。

## Operation 记录

所有会改变书签状态的动作都记录为 operation。链接检测是 evidence，不是 operation。

阶段建议：

- `dead_link_triage`
- `tree_restructure`
- `bookmark_assignment`
- `final_review`

目录变化也用 operation 表示，例如 `create_folder`、`rename_folder`、`move_folder`、`delete_empty_folder`。执行完这些操作后，当前目录树由 CLI replay 生成。

operation 设计原则：

- 分类建议可以大胆，但执行必须保守：只有用户批准的 operation 才写入 `operations.jsonl`。
- 原始 `snapshot.json` 不修改，当前状态永远由 `snapshot.json + operations.jsonl` replay 得到。
- 检测结果、统计结果和 agent 推理是 evidence；只有会改变书签状态的动作才是 operation。
- `dead_link_triage` 阶段优先使用 `mark_bookmark`、`archive_bookmark`、`update_url`，不要默认删除。
- 每个 operation 应包含 `phase`、`type`、目标对象 ID、理由、置信度和可读摘要，方便 WebUI 审核。
- 被拒绝的方案不写入已批准 operation；可以记录在 `decisions.json`，作为后续生成新方案的上下文。
