# 技术文档翻译

## 使用时机

用于翻译技术文档，并让译文符合中文技术文档规范。默认方向是外文技术文档到简体中文；用户明确指定目标语言时，按用户指定语言翻译。

适用内容：

- README、安装说明、迁移指南、变更说明。
- API / CLI 参考、参数说明、配置文件说明。
- PRD、技术规范、架构说明、运维 runbook、故障排查文档。
- Markdown、注释性文档、带命令和配置片段的说明文本。

## 翻译流程

1. 先识别文档角色：教程、README、API/CLI 参考、运维 runbook、规范、PRD、变更说明或故障排查。
2. 判断目标读者：开发者、运维、项目维护者、新用户或决策者。
3. 默认输出完整译文，保留原文 Markdown 结构；只有用户要求审校、术语确认或逐段校对时，才输出原文+译文对照。
4. 翻译时优先保证技术契约准确，再追求中文自然流畅。
5. 翻译完成后做规范检查：结构是否清晰、术语是否一致、步骤是否可执行、示例是否与正文一致。

## 必须保留

不要翻译或改写这些内容，除非用户明确要求：

- 代码块、inline code、命令、参数、路径、文件名、包名、模块名。
- 环境变量、配置键、API 字段、HTTP 方法、状态码、错误码、版本号。
- 产品名、协议名、库名、框架名、专有名词和正式英文缩写。
- Markdown 标题层级、列表层级、表格列、链接地址、引用块和 frontmatter 字段。

链接文本可以翻译，但 URL 不变。代码块里的注释只有在用户明确要求时才翻译；翻译注释也不得改变代码行为。

## 技术文档规范

翻译不只是换语言，还要保持文档可用：

- 标题应准确概括章节内容，不为了文学化而牺牲可扫描性。
- 操作步骤应按“前置条件 -> 操作 -> 预期结果/验证方式”组织。
- CLI/API/配置文档必须保持命令、参数、字段名和正文描述一致。
- 示例命令应能直接复制执行；不要把占位符翻译成自然语言。
- 配置示例应保留原键名和数据结构，只翻译解释文字。
- 故障排查文档应清楚区分症状、原因、验证命令和修复步骤。
- 规范/PRD 文档应保留 MUST/SHOULD/MAY 等约束强度，可译为“必须/应该/可以”，但不要弱化语气。

## 术语策略

- 首次出现的关键术语可使用“中文（English）”或“English（中文）”，后文保持一种写法。
- 行业内常用英文术语可保留英文，例如 API、CLI、token、schema、runtime、fallback。
- 对容易误解的术语，优先保留英文并补中文解释。
- 同一文档内不要混用多个译名；遇到不确定术语，在译文后给出“待确认术语”。

## 联网与查证

默认不联网。只有用户要求查官方术语、当前资料、库/API 文档、版本差异或最新行为时，才按当前环境的文档检索规则查证。查证后在回答中简要说明依据；不要把未查证的猜测写成确定事实。

## 输出检查

交付前检查：

- Markdown 结构、表格、代码块、链接和 frontmatter 没有被破坏。
- 命令、路径、字段名、状态码、配置键与原文一致。
- 术语在全文中保持一致。
- 步骤包含必要的验证方式或预期结果。
- 译文是技术写作风格：清楚、直接、可执行，避免口号化和过度意译。

如果发现原文有明显错误、示例与正文不一致、命令不可执行或术语冲突，不要静默修正；在译文后追加“译注/待确认项”，说明问题和建议。
