# 容器端口暴露与防护速查

## 适用场景

这份文档适合下面这类 Linux 主机：

- 使用 Docker 或 Docker Compose 运行数据库、对象存储、管理面板、通知服务
- 主机本身有公网 IP，或者云安全组 / 防火墙可能放行了容器端口
- 需要判断哪些端口应该保留公网，哪些应该回收到本机、内网或 Tailscale

## 核心结论

- 能不公开的服务，优先不要对公网发布端口
- 数据库、对象存储、管理后台默认不应直接暴露公网
- `只绑定 localhost` 比 `只靠防火墙拦截` 更靠里、更稳妥
- 防火墙适合作为第二层兜底，而不是唯一防线
- 必须公网可达的服务，应最小化端口、加认证、加日志、加自动封禁

## 防火墙关端口 vs 只绑定 localhost

### 防火墙关端口

含义：

- 服务依然监听 `0.0.0.0` 或 `::`
- 网络层在入口处拦掉外部连接

特点：

- 配置简单，适合统一控制整台机器的入站策略
- 可以快速止血，减少被扫后的认证尝试和日志噪音
- 一旦防火墙规则失效、放宽或被误改，服务会立刻重新暴露
- 服务本身仍然处于“可被外部命中”的监听状态

### 只绑定 localhost

含义：

- 服务只监听 `127.0.0.1` 或 `::1`
- 外部主机即使能访问服务器，也连不到这个端口

特点：

- 属于服务级收口，默认更安全
- 即使防火墙配置出错，外部也通常无法直接连入
- 能更彻底减少外部扫描打到应用层
- 可能影响其他容器通过宿主机 IP 或 `host.docker.internal` 访问该服务

### 资源消耗区别

- 两者都能减少公网扫描带来的认证尝试、日志写入和应用层处理
- 只关防火墙端口，服务依然运行并监听，所以基础内存占用基本不变
- 只绑定 `localhost` 通常比单纯防火墙更彻底，因为很多外部连接根本到不了应用层
- 如果目标是明显减少资源占用，收益最大的仍然是停掉不用的容器

## 推荐的防护层级

建议按下面顺序设计：

1. 不需要公网的服务，优先只绑定 `localhost`、内网 IP 或 Tailscale IP
2. 使用 `ufw` / `nftables` 再做一层默认拒绝入站
3. 必须公网开放的 Web 服务，放到反向代理后面并补上认证
4. 必须公网开放且会被持续扫描的服务，增加自动封禁或信誉拦截

## 服务类型与默认策略

### 1. 数据库类

典型服务：

- PostgreSQL
- Qdrant
- Redis
- MongoDB

默认策略：

- 不暴露公网
- 优先只绑定 `127.0.0.1`
- 若需要跨设备访问，优先使用 Tailscale、VPN、SSH 隧道或内网

原因：

- 这类服务的协议面通常不是给互联网直接使用的
- 即使有密码，也会持续受到撞库、弱口令探测和版本指纹扫描

### 2. 对象存储类

典型服务：

- MinIO
- RustFS

默认策略：

- API 端口优先内网或 Tailscale
- Console / 管理台默认不暴露公网

原因：

- 控制台一旦暴露，通常就是高价值管理入口
- API 即使需要跨容器使用，也不一定要直接放在公网

### 3. 管理面板 / 内部工具类

典型服务：

- Beszel
- Gotify
- 各类 Admin、Dashboard、监控后台

默认策略：

- 只给自己或小范围团队访问
- 优先改为 `localhost + Tailscale`
- 若必须公网，至少放在 Caddy / Nginx / Traefik 后面并补认证

### 4. 必须公网可达的协议服务

典型服务：

- RustDesk 服务端
- 对外 API 网关
- 公开网站

默认策略：

- 只开放真正必须的端口
- 用防火墙缩小来源范围或配合自动封禁工具
- 做好日志、速率限制和升级维护

## 一台常见开发机的端口整理示例

下面是一个典型的容器主机整理思路：

| 服务 | 常见端口 | 建议暴露方式 | 原因 |
| --- | --- | --- | --- |
| PostgreSQL | `5432/tcp` | 仅本机 / Tailscale | 数据库不适合直接上公网 |
| Qdrant | `6333/tcp` | 仅本机 / Tailscale | 向量库接口不应裸露公网 |
| RustFS API | `9000/tcp` | 内网 / Tailscale / 反向代理后按需开放 | API 通常只给应用或受控客户端 |
| RustFS Console | `9001/tcp` | 仅本机 / Tailscale | 管理后台优先收口 |
| Beszel | `8090/tcp` | 仅本机 / Tailscale | 监控面板属于管理面 |
| Gotify | `30080/tcp` | 仅本机 / Tailscale | 通知后台不建议直接裸露公网 |
| RustDesk HBBS | `21115/tcp`、`21116/tcp`、`21116/udp` | 按需公网保留 | 客户端连接所需 |
| RustDesk HBBR | `21117/tcp` | 按需公网保留 | 中继所需 |
| RustDesk Web / Pro 相关 | `21114/tcp`、`21118/tcp`、`21119/tcp` | 不用则关闭 | 不是所有部署都需要 |

## Docker Compose 配置建议

### 只绑定 localhost

```yaml
services:
  postgre:
    ports:
      - "127.0.0.1:5432:5432"
```

适用场景：

- 只需要宿主机本地访问
- 通过 SSH 隧道或 Tailscale 访问
- 不需要让公网或其他宿主机直接连接

### 直接发布所有网卡

```yaml
services:
  postgre:
    ports:
      - "5432:5432"
```

风险：

- 会绑定到 `0.0.0.0` / `::`
- 如果安全组或防火墙放行，公网会直接打到这个服务

## 现成工具怎么选

### `ufw` / `nftables`

适合：

- 先把机器收口
- 用最小成本减少无意义的公网访问

优点：

- 系统级统一治理
- 对所有进程和容器都有效

注意：

- 这是兜底层，不替代服务自身的收口

### `Tailscale`

适合：

- 只想给自己、自己的设备或小团队访问后台
- 想避免把数据库、管理面板直接放到公网

优点：

- 部署轻
- 不需要额外暴露业务端口给公网
- 很适合 `PostgreSQL`、`Qdrant`、`Beszel`、`Gotify`、对象存储控制台

### `Caddy`

适合：

- 某些 Web 服务必须公开
- 需要 HTTPS、基础认证、统一入口

优点：

- 配置简单
- 适合给管理面板或内部工具补一层反向代理和认证

### `CrowdSec`

适合：

- 服务必须面对公网
- 希望自动封禁高频扫描、爆破和恶意 IP

优点：

- 比手工维护黑名单更自动化
- 适合补在长期暴露公网的主机上

### `Fail2ban`

适合：

- 某些服务日志格式明确、规则较稳定
- 想快速对 SSH、部分 Web / 数据库日志做基于日志的封禁

注意：

- 对现代多容器环境不如 `CrowdSec` 灵活，但仍然是低门槛方案

## 一套实用的落地顺序

如果目标是“尽量少折腾，但显著降风险”，可以按这个顺序做：

1. 先盘点当前所有对外暴露端口
2. 区分“必须公网”和“只给自己访问”的服务
3. 把数据库、对象存储 Console、管理后台改成 `localhost` 或 Tailscale
4. 用 `ufw` / `nftables` 对公网入站做默认拒绝
5. 对必须公网的服务保留最小端口集合
6. 对公开 Web 服务增加 Caddy 反代和认证
7. 对长期暴露公网的服务补 `CrowdSec` 或 `Fail2ban`

## 快速判断规则

可以用下面这组问题快速决定一个端口该不该公开：

- 这个服务是否必须被陌生公网客户端直接访问？
- 这个端口是否是数据库、存储、管理面板，而不是公开业务入口？
- 这个服务是否能改成通过 Tailscale、VPN、SSH 隧道访问？
- 这个服务如果被持续扫描，是否会产生额外日志、认证尝试或误报？
- 这个服务是否真的需要所有当前开放的端口？

如果前两个问题的答案分别是“否”和“是”，通常就不该直接暴露公网。

## 参考命令

查看当前监听端口：

```bash
ss -ltnup
```

查看 Docker 端口映射：

```bash
docker ps --format 'table {{.Names}}\t{{.Image}}\t{{.Ports}}'
docker inspect --format '{{json .HostConfig.PortBindings}}' <container>
```

查看某个服务是否绑定到所有网卡：

```bash
ss -ltnp | rg ':5432\b'
```
