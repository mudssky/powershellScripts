# 编程目录结构通用原则

## 先读证据

整理代码项目结构前，先识别：

- 语言、框架、运行时和包管理器。
- 项目 manifest：例如 `package.json`、`pyproject.toml`、`go.mod`、`Cargo.toml`、`pom.xml`、`build.gradle`、`.sln`。
- 应用入口、库导出入口、CLI 入口、配置加载入口。
- 构建、测试、格式化、lint、CI 和发布入口。
- import、相对路径、文档链接、脚本参数和配置字段。
- 生成产物、缓存、本机私有配置和 secret。

框架项目必须先查框架官方文档和当前项目约定。语言通用结构只能作为补充，不能覆盖框架目录结构。

## 通用方法

### 入口优先

入口文件应薄，负责参数解析、启动、依赖装配和错误边界。复杂业务逻辑下沉到模块、服务、domain 或 core。

常见入口：

- CLI：`main`、`cli`、`cmd`、`bin`。
- API：router、controller、handler。
- 应用：app、server、bootstrap。
- 库：公开导出文件或 package exports。

### 包/工作区边界优先

先识别包边界，再移动文件。不要把某个包的私有规则扩散到根目录，也不要把根级配置塞进子包。

常见边界：

- monorepo workspace。
- 单语言 module/package/project。
- 独立部署单元。
- 共享库和应用之间的边界。

### 功能/领域垂直切片

适合产品功能快速演进的应用代码。把同一功能的 handler、service、model、schema、test、文档靠近组织。

优点：功能改动集中，容易删除或迁移。

风险：跨功能共享逻辑容易重复，需要共享能力抽取规则。

### 架构层分层

适合边界清晰、复用多、团队分工明确的系统。常见层包括 `api`、`service`、`domain`、`adapter`、`infra`、`ui`。

优点：职责清晰，跨功能复用方便。

风险：小项目过早分层会增加跳转成本。

### Ports/Adapters

核心逻辑不依赖数据库、HTTP、文件系统、云服务或模型供应商；外部依赖通过 adapter 接入。

适合：

- 需要替换外部服务的工具。
- 多供应商、多存储、多协议项目。
- 测试需要隔离外部副作用的项目。

### 共享能力抽取

相似逻辑出现多处时，考虑抽到共享模块、公共包、core 或 utils。

抽取条件：

- 重复出现三次或更多。
- 逻辑复杂且容易出错。
- 多个模块需要同一行为保持一致。

不要为了单次使用或一行简单表达过早抽象。

### 测试位置

小模块可以 colocate test，让行为和实现一起演进。大型项目可以用镜像目录或根级集成测试。

选择标准：

- 单元测试跟随被测模块。
- 集成测试跟随系统边界。
- 端到端测试跟随用户入口或部署入口。

### 配置分层

把默认配置、可提交示例、本机私有覆盖、环境变量和 CLI 参数分清楚。

常见优先级：

```text
CLI 参数 > 环境变量 > 本机私有配置 > 可提交配置 > 内置默认值
```

不要把真实 secret、token、连接串写入可提交文件。

### 源码和产物分离

源码真相、构建产物、可执行 shim、报告、缓存要有明确边界。

规则：

- `src` 或语言约定目录保存源码真相。
- `dist`、`build`、`target`、`bin`、coverage、cache 通常是产物或入口，不要反向编辑生成物。
- 如果必须提交生成入口，文档要说明生成来源和验证方式。

### 文档位置

API、模块说明和设计约束可以靠近代码。跨模块知识、长期手册和总索引适合放集中 docs。

分类时按读者最可能查找的入口放置，并用交叉链接补充次要路径。

## 输出模板

整理代码项目结构时，默认用：

| 对象 | 当前位置/结构 | 建议归属 | 操作 | 方法论依据 | 风险 | 验证方式 |
|---|---|---|---|---|---|---|

`验证方式` 至少说明要搜索哪些旧路径、运行哪些测试或检查哪些 manifest。
