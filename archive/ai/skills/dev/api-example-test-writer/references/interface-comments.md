# 接口注释规范

## 目标

`.http` 文件既是测试，也是活文档。每个请求块都要让读者知道它为什么存在、依赖什么、验证什么，而不是只看到一段可执行请求。

## 覆盖范围

- 所有请求块都写注释。
- 初始化脚本块、变量准备块虽然不是接口，也要说明它在流程里的作用。
- `manual` 请求可以更短，但必须说明适用场景、为什么不能进 CI、是否有敏感风险。

## 注释维度设计

常用的三个点对应一个测试请求的最小判断链：

- `说明`：目标。说明这个请求在证明什么，回答“为什么要跑它”。
- `前置`：条件。说明环境、账号、权限、数据状态、`@ref` 依赖等，回答“失败时先怀疑哪里”。
- `断言`：判定标准。说明为什么检查这些状态或字段，回答“什么结果算通过”。

它不是固定三字段模板。所有请求块至少要有 `说明`；关键自动化请求、跨接口依赖、异步流程和负例请求再补 `前置`、`断言`。手动或高风险请求优先补 `风险`，初始化脚本块可以使用 `设计` 解释状态准备方式。

## 推荐注释结构

```http
###
# 说明：验证登录接口能为测试账号签发访问令牌，后续请求通过 @ref 复用该结果。
# 前置：当前环境的 username/password 指向可登录的非生产测试账号。
# 断言：只检查 token 存在，不校验具体 token 内容，避免把实现细节写进测试。
# @name login
# @tag test
POST {{baseUrl}}/api/auth/login
Content-Type: application/json
```

按需保留这些信息，不要求每块都写满：

- `说明`：接口或流程的业务目的。
- `前置`：认证、数据状态、环境、账号、租户、外部依赖。
- `参数`：只解释不直观或有业务约束的关键参数。
- `断言`：说明为什么断言这些字段或状态。
- `风险`：说明手动请求、写操作、生产禁用、幂等性或数据清理要求。

## 写法原则

- 写业务意图，不翻译语法。
- 写维护者需要知道的上下文，不复述 URL、HTTP 方法和字段名。
- 先短后准；一个请求块通常 2 到 4 行注释足够。
- 注释要跟断言同步更新。断言变了，断言意图也要变。
- 对跨接口流程，说明 `@ref` 依赖和响应变量复用关系。
- 对异步流程，说明轮询终止条件、失败状态和超时边界。

## 好例子

```http
###
# 说明：读取当前登录用户信息，用来确认 login 令牌能访问受保护接口。
# 前置：依赖 login 请求成功；该请求失败通常说明认证链路或测试账号异常。
# 断言：只验证稳定身份字段存在，避免测试绑定展示字段。
# @name me
# @tag test
# @ref login
GET {{baseUrl}}/api/me
Authorization: Bearer {{login.response.body.$.data.token}}
```

## 坏例子

```http
###
# 发送 GET 请求到 /api/me。
# 请求头 Authorization 是 Bearer token。
# @name me
GET {{baseUrl}}/api/me
Authorization: Bearer {{token}}
```

问题：这只是复述请求本身，没有解释为什么请求、依赖什么、失败时看哪里。

## 手动请求

```http
###
# 说明：本地调试登录失败时使用，不进入 CI。
# 风险：账号和密码必须来自本机 env，禁止把真实凭据写进请求体。
# @name login-manual
# @tag manual
POST {{baseUrl}}/api/auth/login
```

`manual` 请求不等于无规范。它可以没有完整断言，但必须清楚标出边界。
